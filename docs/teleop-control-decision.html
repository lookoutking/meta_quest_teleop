VR 手把遙操作雙手臂機器人：控制方法選擇

情境：T 型架雙手臂 · MPC 控制 · 模擬環境 手把：Meta Quest（6-DoF）
系統前提：T 型架雙手臂，臂長對標人體，MPC 接受 end-effector pose 目標。 VR 手把透過 ADB 傳輸姿態資料，座標系為 OpenXR（已可轉換至 ROS）。 目前 APK 未發布頭盔（HMD）pose。

方法詳細分析
A．絕對位置模式
VR 手把世界坐標 → 機器手臂 end-effector 世界坐標（直接對應）
固定架適用

優點

  • 操作最直覺，手在哪機器手就在哪
  • 臂長對標人體時 scale factor ≈ 1，不需縮放
  • 模擬環境中 URDF frame 位置固定，對應關係只需建立一次
  • 不需 HMD pose

限制

  • 依賴 meta_world 原點穩定，每次開機位置不同需重新對齊
  • T 型架若移動，需重新建立坐標系對應
  • 容易將機器手臂逼進奇異點或超出可達範圍
實作重點：建立固定轉換矩陣 T_meta_to_robot，對應 meta_world 原點與機器人基座 frame。模擬環境中可從 URDF 直接推算，毋需實體標定。
B．相對位移模式（World Frame Delta）
計算手把每 tick 的 ΔT，疊加到機器手臂當前 end-effector pose
通用備選

優點

  • 不需原點校正，T 型架移動不影響操作
  • 可加入 Clutch（按住 grip 才追蹤）隨時重置手部位置
  • 支援 scaling：小幅移動 → 精細控制

限制

  • 操作者感覺是「控制移動量」而非「控制位置」，認知負擔較高
  • 手把姿態與機器手臂姿態不一致時，移動方向可能不直覺
  • 長時間累積誤差，需定期重置
實作重點：需要定義 Engage 機制（如 grip trigger 按住才追蹤 delta）以支援手部重置。需一次性旋轉對齊 meta_world 與 robot world 的軸向。
C．頭盔相對姿態模式
發布手把相對 HMD 的 pose，以操作者視角為控制基準
長期最佳

優點

  • 完全不需要任何坐標系校正
  • 「往前推」永遠對應機器手臂往前，最符合人體直覺
  • T 型架移動、meta_world 原點改變皆不影響
  • 操作者轉向，控制方向自動跟著轉

限制

  • 需要 APK 發布 HMD pose（目前尚未實作）
  • 需在 reader.py 新增 get_head_transform_ros()
  • 操作者需面對機器人站立，偏轉角度過大時直覺性下降
需補實作：① OculusTeleop.cpp 將 HeadPose 加入 logcat 輸出；② reader.py 解析 "h" key 並新增 get_head_transform_ros()；③ ROS2 node 計算 T_hand_rel = T_hmd⁻¹ @ T_hand。
D．速度控制模式（Twist）
手把傾斜角度決定 end-effector 移動速度與方向
MPC 整合佳

優點

  • 對延遲最不敏感，輸入是方向意圖而非精確位置
  • 直接以 twist 指令餵給 MPC，整合最簡單
  • 容易加入速度限制做安全保護

限制

  • 學習曲線最高，操作不直覺
  • 靜止需手把保持在基準姿態，長時間易疲勞
  • 精確定位困難，不適合需要精細操作的任務
適合場景：大範圍快速移動、安全測試階段、或 MPC 已有強健軌跡跟蹤能力時作為輔助模式。
建議路徑

分階段實作建議

短期（模擬驗證）：以方法 A（絕對位置）先行。模擬環境中 URDF frame 固定， 從 tf2 查詢手臂基座位置建立 T_meta_to_robot，可快速驗證 MPC 接收 end-effector 目標的整體流程， 不需修改 APK。

中期（補齊 HMD pose）：補實作 APK HMD pose 輸出與 reader.py 對應方法， 切換至方法 C（頭盔相對姿態）。這是對 T 型架可移動情境最完整的解， 也是實體部署時最不需要重新校正的方案。

操作體驗補強：無論選哪種方法，建議加入 Clutch 機制（grip trigger engage） 讓操作者可隨時凍結機器手臂並重置手部位置，減少超出可達範圍的情況。
